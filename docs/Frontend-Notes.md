# Frontend Notes for Ledgerful/Ledgerful

This document serves as the reverse-contract for the `ledgerful-frontend` repository. It captures backend architectural decisions, upcoming API changes, feature flags, and technical constraints that the frontend team needs to be aware of. 

When backend developers modify `/api/*` routes, change JSON shapes, or introduce new config-gated features, they must document those changes here so the frontend team can adapt.

## 1. Backend Constraints & Philosophy
- **Stateless Daemon:** The Ledgerful daemon does not persist session state between restarts for local users. The `?token=` mechanism is purely ephemeral.
- **Honest Zeros:** If the backend does not yet support collecting certain data (e.g., `testsRun` or `flakes` for legacy transactions), the API will return `0`. The frontend should display these as `0` or hide the metric entirely, rather than attempting to interpolate or mock missing data in production.
- **Snake Case vs Camel Case:** The backend is written in Rust and naturally serializes structs as `snake_case`. The frontend is responsible for mapping these to `camelCase` in its API client layer.

## 2. Feature Flags & Configuration Gating
The backend relies heavily on `config.toml` to gate features. The frontend should handle these gracefully:
- `coverage.enabled`: If this is false, endpoints like `services diff` or `/api/security/boundaries` may return empty states. The frontend should prompt the user with the proactive config hint (e.g., `ledgerful config set coverage.enabled=true`) rather than treating it as an error.
- `intent.require_signing`: If false, the ledger API may omit signatures. The frontend compliance UI must account for unsigned transactions without crashing.

## 3. Upcoming API Breaking Changes / Roadmap
*(Backend developers: Add planned changes here before they merge to main)*
- **Track DX5 (Intent-Driven Context Pruning):** Changes to `ask` query routing may subtly alter the context chunks returned for global architecture questions.
- **Track DX6 (Auto-Scan):** The `/api/changes` endpoint might perform a dynamic in-memory scan rather than reading `latest-impact.json` if auto-scan is enabled. Response times may increase slightly.
- **Track DX7 (Global Proactive Config Enablement):** `ledgerful deploy impact --json` now emits the standard empty-state payload (`{ results: [], emptyReason: "disabledByConfig", message: "..." }`) when `coverage.enabled` or `coverage.deploy.enabled` is `false`, instead of a bare `[]` array. No new `emptyReason` enum value was added — `disabledByConfig` already existed. The `deploy impact` command now reads from the config-gated impact packet (`deploy_manifest_changes`) rather than a never-populated `deploy_manifests` table, so its empty state is now genuinely config-gated. Frontends should render the `message` field (which contains the copy-pasteable `ledgerful config set <key>=true` hint) when `emptyReason` is `disabledByConfig`. **JSON shape asymmetry (standard `format_json_empty_state` contract):** `deploy impact --json` returns a bare JSON array of manifests (`[{ "path": ... }, ...]`) when populated, but an object (`{ "results": [], "emptyReason": "disabledByConfig"|"noMatches", "message": "..." }`) when empty. This array-when-populated / object-when-empty asymmetry is the EXISTING `format_json_empty_state` contract shared by ALL sparse CLI surfaces (`data-models impact`, `endpoints`, `deploy impact`), not a deploy-specific quirk. JSON consumers of these surfaces already handle both shapes by design — do not assume the top-level value is always an array or always an object. Changing `deploy impact` alone to a different envelope would be inconsistent with the other surfaces.
- **Future Track (Diff Stats):** Future schema migrations will add `additions` and `deletions` to the `changed_files` table. Once released, the `/api/ledger/:txId` payload will start returning non-zero values for these fields.
- **Track (Hotspot Data Shape):** The `GET /api/hotspots` endpoint returns a `HotspotResponse` DTO (Track TA29) with the exact shape the frontend expects: `id`, `filePath`, `riskLevel` (CRITICAL/HIGH/MEDIUM/LOW), `riskScore` (=displayScore, log scale), `lastTouchedAt` (ISO-8601 or null), `contributor` (author name or null), `changeCount` (frequency.round(), min 1), `rank` (1-based). Backward-compat fields (`displayScore`, `score`, `complexity`, `frequency`, `centrality`) are included temporarily until the frontend PR merges. `lastTouchedAt` and `contributor` are persisted in `project_files` during indexing (Track TA30, migration m47) and read directly from SQLite; a 5-minute TTL cache is the fallback for DBs without the migration. Risk-level thresholds use the log scale (4.0/3.0/2.0), NOT 0-100.
- **Track E1 (Verification Dashboard APIs):** The three read-only verification endpoints the frontend previously mocked now have real backend implementations (gated by the default `web` feature):
  - `GET /api/verify/health` → `{ "status": "HEALTHY"|"DEGRADED"|"FAILING", "lastRunAt": string|null, "message"?: string }` (camelCase). `DEGRADED` covers both "no runs recorded" (`lastRunAt` is `null`, `message` explains) and "latest run older than 7 days and passing" (a stale failing run stays `FAILING`). Never returns 404 for empty state — it returns 200 with `DEGRADED`.
  - `GET /api/verify/history?days=30` → bare JSON array of `{ "date": "YYYY-MM-DD", "passed": number, "failed": number }` (camelCase), sorted ascending by date. Default window 30 days, accepts up to 365. Dates with zero runs are **omitted** (deterministic: only dates with ≥1 run appear). Aggregated from `verification_runs` by `DATE(timestamp)`.
  - `GET /api/verify/steps` → bare JSON array of `{ "id": string, "name": string, "lastRunAt": string, "averageDurationMs": number, "passRatePercent": number, "recentFailures": number }` (camelCase), sorted ascending by step command. `id` is the step's stable identifier (its `command` — the verify plan step has no separate id field; `verification_results` only stores `command`). `name` is the friendly label: the step's `description` from the most recent `verification_runs.plan_json` (a serialized `VerificationPlan`) that contains a step with that command, falling back to `command` when no `plan_json` is available, fails to parse, or has no matching step. `passRatePercent` is rounded to 2 decimal places. `recentFailures` counts failures within the last 10 verification runs. Source: `verification_results` joined to `verification_runs`.
  - These endpoints follow the existing web API pattern: token-gated under `/api`, `tokio::task::spawn_blocking` for SQLite reads, `WebError` for failures. The casing is camelCase via `#[serde(rename_all = "camelCase")]`, consistent with the existing `ChangeResponse` sibling (note: §1's general "snake_case" statement does not apply to these endpoints — they match the camelCase `ChangeResponse`/`EmptyReason` siblings per the consistency rule).
- **Track E2 (Compliance Dashboard APIs):** The two read-only compliance endpoints the frontend previously mocked now have real backend implementations (gated by the default `web` feature):
  - `GET /api/compliance/summary` → `{ "totalSigned": number, "validityPercent": number, "lastAuditAt": string|null, "hotspotDeltaPercent": number }` (camelCase). `totalSigned` counts ledger entries with BOTH `signature` and `public_key` present (signed rows, regardless of whether the signature currently verifies). `validityPercent` is `validCount / totalEntries * 100` rounded to 2dp where `validCount` is the number of entries whose Ed25519 signature verifies against the stored public key; `0.0` when there are no entries. `lastAuditAt` is the most recent `committed_at` among entries classified `VALID` (the last verifiably-signed event); `null` when there are no `VALID` entries / no entries — this field is intentionally NOT `skip_serializing_if`-gated, so it serializes as `null` (not omitted) in the empty state (same contract as E1's `lastRunAt`). `hotspotDeltaPercent` is the percent change in total hotspot count between the two most recent `hotspot_history` snapshots: `((newerTotal - olderTotal) / olderTotal) * 100`, 2dp; `0.0` when there are fewer than 2 snapshots; `100.0` when `olderTotal == 0` and `newerTotal > 0`. Never returns 404 for empty state — it returns 200 with `{ "totalSigned": 0, "validityPercent": 0.0, "lastAuditAt": null, "hotspotDeltaPercent": 0.0 }`.
  - `GET /api/compliance/signatures` → bare JSON array of `{ "txId": string, "entity": string, "summary": string, "committedAt": string, "status": "VALID"|"INVALID"|"SKIPPED", "category": string }` (camelCase), sorted by `committedAt` DESC, bounded to the most recent 100 entries. `status`: `(signature, publicKey)` present → `verify_signature` → `VALID` if it verifies else `INVALID`; absent → `INVALID` when `intent.require_signing=true` (the CLI labels this `UNSIGNED`; the dashboard status set is `{VALID,INVALID,SKIPPED}` so `UNSIGNED` maps to `INVALID`), else `SKIPPED`. `category` is the `Category` enum's string form (e.g. `"FEATURE"`). Empty state → `[]` (200, not an error).
  - These endpoints follow the same web API pattern as the E1 verify endpoints (token-gated, `spawn_blocking` SQLite reads, camelCase via `#[serde(rename_all = "camelCase")]`). Classification mirrors `verify_ledger_signatures` in `src/commands/verify.rs`. The signed payload is `tx_id\ncategory\nsummary\nreason\ncommitted_at` (see `sign_ledger_entry` in `src/ledger/crypto.rs`).
- **Track E3 (SOC2 Evidence Export API):** The export endpoint the frontend previously mocked now has a real backend implementation (gated by the default `web` feature):
  - `GET /api/compliance/export` → binary download. Response headers: `Content-Type: application/zip`, `Content-Disposition: attachment; filename="ledgerful-soc2-evidence.zip"`. The body is a `.zip` generated on the fly; the frontend should trigger a browser download (it is NOT JSON — do not attempt to parse it as JSON). Token-gated under `/api` like the other compliance endpoints. Empty/no-DB state still returns 200 with a valid zip (header-only CSVs, no `adr/` files); it never returns 404 for empty state.
  - **Zip layout:** `manifest.json` (cryptographic hashes + sizes of all other files, plus `generatedAt` and `entryCount`, camelCase JSON, `files` array sorted by `name` ASC for determinism), `manifest.sig` (raw 64-byte Ed25519 signature over the `manifest.json` bytes), `manifest.pub` (raw 32-byte Ed25519 verifying key), `ledger.csv` (all committed transactional provenance records, RFC 4180 CSV, columns `tx_id,category,entity,change_type,summary,reason,committed_at,signed,signature`, rows sorted by `committed_at` ASC), `verification_history.csv` (CI gate pass/fail records, columns `run_timestamp,overall_pass,command,exit_code,duration_ms`, ordered by `run_timestamp DESC` then `command ASC`, header-only when there are no runs), and `adr/*.md` (generated MADR-format Architectural Decision Records, one file per ADR ledger entry, filename `0001-<slugified-summary>.md`).
  - **`manifest.json` shape:** `{ "generatedAt": "<RFC3339>", "files": [{ "name": "ledger.csv", "sha256": "<hex>", "size": <u64> }, ...], "entryCount": <u64> }`. `entryCount` is the total number of ledger entries in `ledger.csv` (this includes ADR ledger entries, since ADRs are entries where `entry_type='ARCHITECTURE'` OR `is_breaking=1` (matches `ledgerful ledger adr list`)). `manifest.json`, `manifest.sig`, and `manifest.pub` are intentionally NOT listed in the `files` array — they ARE the manifest + signature.
  - **Tamper-evidence contract (offline verification):** To validate an export bundle offline: (1) for each entry in `manifest.json`'s `files`, re-SHA-256 the corresponding file's bytes from the zip and compare against the manifest's `sha256` — any mismatch means the file was modified after the manifest was generated; (2) read `manifest.sig` (raw 64 bytes) and `manifest.pub` (raw 32 bytes), reconstruct `ed25519_dalek::Signature::from_bytes` and `VerifyingKey::from_bytes`, and call `verifying_key.verify(manifest_json_bytes, &signature)` — this proves the manifest itself was produced by the holder of the repo's Ed25519 private key and has not been altered. The export reuses the repo's existing keypair (`crate::ledger::crypto::get_or_create_keys`, the same keypair `verify --signatures` uses, stored in `~/.ledgerful/keys/`), so the existing offline verifier can validate the export. `manifest.sig` and `manifest.pub` are raw bytes (not hex) to keep the `Signature::from_bytes` path direct.
- **Track TA31 (Stale Project Auto-Sync + Migration Prompt):** Federation/dashboard changes:
  - `GET /api/projects` (`ProjectResponse`): added `validation_warnings: string[]` (snake_case on the wire — this endpoint was never migrated to camelCase). Always present; empty for the local/root project and for fully-valid siblings. Non-empty entries are deterministic, human-readable strings (e.g. `"Invalid schema: ledger entity must not be empty"`) describing data-quality problems in a sibling's `schema.json` that did NOT prevent discovery — render these as a "needs attention" badge/tooltip rather than hiding the project. This is additive; existing consumers that ignore unknown fields are unaffected.
  - **Loosened sibling validation:** a sibling whose `schema.json` has a data-quality problem (e.g. an empty ledger `entity` — the AI-Brains case) is now still discovered and returned by `/api/projects`, instead of being silently omitted. Only security violations (path traversal, absolute paths) or an unsupported `schema_version` still cause a sibling to be omitted entirely.
  - **Empty `entity` fallback:** any API response that serializes a ledger entry's `entity` field (`LedgerEntryResponse` via `GET /api/ledger*`, `ComplianceSignatureEntry` via `GET /api/compliance/signatures`) now renders `"(uncategorized)"` instead of an empty string when the underlying entity is blank. This is a presentation-only fallback applied at JSON-serialization time — the raw stored value (SQLite, schema.json) is untouched, so do not rely on this string for filtering/grouping logic that needs to distinguish "genuinely uncategorized" from a future literal entity named `"(uncategorized)"`.
  - **`schema.json` version stamping:** the on-disk federation schema (not a `/api/*` payload, but consumed by sibling backends) gained two new fields: `generated_at` (RFC3339 timestamp of last `federate export`) and `binary_version` (`CARGO_PKG_VERSION` of the exporting binary). Both default to `""` via `#[serde(default)]` for old schema.json files — frontend code does not read schema.json directly, this is backend-to-backend, noted here only for completeness.
  - **New config flag `[federation].auto_sync_siblings`** (default `false`): when enabled, `ledgerful federate scan` will auto-regenerate a sibling's missing/stale `schema.json` by sequentially shelling out to `federate export` in that sibling's directory. No new API surface; mentioned here because it changes how often/reliably siblings appear in `/api/projects` over time in installs that opt in.
  - **DB migration prompt (CLI-only, no API impact):** opening an existing (pre-TA31) `ledger.db` with a newer `ledgerful` binary now either interactively prompts `Proceed? [Y/n]` on stderr, or — in non-interactive contexts (CI, git hooks, piped stdin) — silently proceeds but prints `[INFO] Ledgerful database auto-migrated from v{old} to v{new}` to stderr. A brand-new project (no pre-existing `ledger.db`) never triggers this. Purely a CLI/daemon-startup concern; does not change any `/api/*` response shape.

## 4. Telemetry and Analytics
- The backend controls the primary telemetry opt-in (`ledgerful usage`). If the frontend introduces its own telemetry, it MUST respect the same local `.ledgerful/config.toml` opt-out settings and use the official Supabase ingest endpoint (`https://scmxtnjqqklvcwyeouvj.supabase.co/functions/v1/telemetry-ingest`).

## 5. Security & Isolation
- Do not expect the local backend to implement SAML/OIDC or role-based access control (RBAC). Those features belong to the future SaaS wrapper (Priority #9/10). The local frontend should not build UI elements that assume these features exist in the local daemon.
